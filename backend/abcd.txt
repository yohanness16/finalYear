Yes, FastAPI has excellent native support for WebSockets, and using Redis with it is a standard industry practice for real-time applications like yours.
1. WebSockets in FastAPI

FastAPI is built on Starlette, which provides a very efficient WebSocket class. You don't need external libraries for basic functionality; it is handled asynchronously by default, which is perfect for streaming GPS data.

    How it works: You define a websocket_endpoint that accepts a connection. Once connected, you can use an infinite loop to send or receive data without the overhead of traditional HTTP requests.

    Winner for your project: Since your project requires a "high-speed data bus" for constant GPS and density metadata, WebSockets will provide much lower latency than standard API polling.

2. Using Redis with FastAPI

Since you mentioned you are avoiding Firebase, Redis is the perfect "winner" for your specific use case for two main reasons:

    Pub/Sub (Real-time Messaging): Redis has a "Publish/Subscribe" (Pub/Sub) feature. Your backend can "Publish" a bus's new location to a specific "channel" (e.g., route_number_5), and all users' mobile apps "Subscribed" to that channel will get the update instantly.

    Geospatial Indexing: Redis has built-in commands like GEOADD and GEODIST. You can store the latitude and longitude of buses directly in Redis and quickly calculate which bus is closest to a user's terminal.

3. Proposed Backend Flow (No-Firebase Architecture)

Here is how your FastAPI + WebSocket + Redis stack would handle the data:

    Ingestion: Your ESP32-CAM sends GPS and density data via the SIM7600 (using MQTT or HTTP) to your FastAPI backend.

Storage: FastAPI saves the latest coordinates in Redis (using a TTL so old data expires automatically).

Distribution: FastAPI "Publishes" this update to a Redis channel.

Delivery: The WebSocket server picks up the message from the Redis channel and "Pushes" it to all connected Flutter mobile apps.

Why this is better for you than Node.js:

Because your project involves OpenCV for passenger counting, staying in the Python ecosystem (FastAPI) allows you to use the same language for your real-time messaging AND your computer vision logic. This avoids the complexity of trying to call Python vision scripts from a Node.js backend.

1. The Haversine Formula (app/services/eta_calc.py)

You'll use this to calculate the distance between the bus and the next stop to provide accurate ETAs.
Python

import math

def calculate_distance(lat1: float, lon1: float, lat2: float, lon2: float) -> float:
    """Calculates the distance in meters between two GPS points."""
    R = 6371000  # Earth radius in meters
    phi1, phi2 = math.radians(lat1), math.radians(lat2)
    dphi = math.radians(lat2 - lat1)
    dlambda = math.radians(lon2 - lon1)

    a = math.sin(dphi/2)**2 + \
        math.cos(phi1) * math.cos(phi2) * math.sin(dlambda/2)**2
    
    return 2 * R * math.atan2(math.sqrt(a), math.sqrt(1 - a))

2. WebSocket Manager (app/services/websocket.py)

from fastapi import WebSocket
from typing import List

class ConnectionManager:
    def __init__(self):
        self.active_connections: List[WebSocket] = []

    async def connect(self, websocket: WebSocket):
        await websocket.accept()
        self.active_connections.append(websocket)

    def disconnect(self, websocket: WebSocket):
        self.active_connections.remove(websocket)

    async def broadcast(self, message: dict):
        for connection in self.active_connections:
            await connection.send_json(message)

from datetime import datetime

def get_time_multiplier():
    now = datetime.now().time()
    
    # Morning Peak: 7:00 to 9:30
    if time(7, 0) <= now <= time(9, 30):
        return 1.5  # Adds 50% more time to stops

        return 1.8  # Adds 80% more time (boarding + city traffic)

import cv2
import numpy as np

def estimate_density(image_bytes):
    # Convert bytes to OpenCV image
    nparr = np.frombuffer(image_bytes, np.uint8)
    img = cv2.imdecode(nparr, cv2.IMREAD_COLOR)
    
    # 1. Convert to Grayscale
    gray = cv2.cvtColor(img, cv2.COLOR_BGR2GRAY)
    
    # 2. Apply Thresholding (detecting dark shapes/people against bus interior)
    _, thresh = cv2.threshold(gray, 120, 255, cv2.THRESH_BINARY_INV)
    
    # 3. Calculate percentage of "filled" pixels
    height, width = thresh.shape
    total_pixels = height * width
    filled_pixels = cv2.countNonZero(thresh)
    density_percentage = (filled_pixels / total_pixels) * 100
    
    # 4. Map to Levels
    if density_percentage < 30: return 0  # Low
    if density_percentage < 70: return 1  # Medium
    return 2 # High
